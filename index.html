OpenClaw App Company

GitHub Pages 向け軽量 Web アプリを自律的に量産する仮想開発会社パイプライン

Overview

市場調査からリリース・改善まで、エージェントが各フェーズを順番に担当する Python 製オーケストレータ。Codex CLI に実装を委任し、 GitHub Pages へ静的 Web アプリを継続的にデプロイする。

各エージェントは artifacts/ 以下に成果物を出力し、 state/company_state.json でパイプライン状態を管理する。

Phases

research → product → design → prompt → release → improvement
research
市場調査・競合分析・アイデアスコアリング・CEO 意思決定
product
プロダクト企画・UX シナリオ・PRD 作成
design
アーキテクチャ・フロントエンド設計・タスク分解
prompt
Codex CLI 向けプロンプト生成
release
QA 確認・GitHub Pages リリース判定
improvement
次スプリント計画・改善ループ選択

Release Outcomes

release フェーズは QA 成果物の内容によって2通りの結果になる。

Release OK
browser_test_report.md と design_review.md が ともに OK / PASS を含む場合。
improvement は normal モードで動作し、 次スプリントの改善案を生成する。
Release Provisional
どちらかが stub / placeholder のまま未更新の場合。 パイプラインは継続するが公開はブロックされる。
improvement は qa_remediation モードで動作し、 修正すべき QA 項目を列挙する。

State Fields

フィールド 意味 例
current_phase 最後に完了したフェーズ improvement
next_action 今回のパイプライン実行結果 pipeline-complete
followup_action 次ランまたは人手で行うべき実務アクション start-task-breakdown / start-qa-remediation

next_action は「この run が何で終わったか」、 followup_action は「次に何をすべきか」を示す。 両者を分けることで、Provisional 時の pipeline-complete / start-qa-remediation という並びが機械にも人にも読みやすくなる。
